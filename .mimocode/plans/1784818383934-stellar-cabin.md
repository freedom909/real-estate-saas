# Audit Subgraph Bug Fixes

## Summary
Fix 8 identified bugs in the audit subgraph, ordered by severity. All changes are minimal and independently verifiable.

---

## Fix 1: recordAuditLog bypasses service layer (HIGH)

**File:** `src/subgraphs/audit/resolvers/audit.resolver.ts`

**Problem:** The resolver calls `AuditModel.create()` directly, bypassing the service and repository layers. This breaks the DI architecture and makes `AuditService.createLog()` dead code.

**Current code (line 28-34):**
```ts
return AuditModel.create({
  action: args.action,
  userId: args.userId,
  resourceId: args.resourceId,
  metadata: args.metadata,
  timestamp: new Date(),
});
```

**Fix:** Route through the service. Also add `metadata` support to `AuditService.createLog()`.

**Steps:**
1. In `audit.resolver.ts`, replace `AuditModel.create(...)` with `service.createLog(...)` via `container.resolve(AuditService)`
2. In `audit.service.ts`, update `createLog()` to accept and pass through `metadata`
3. Remove the `AuditModel` import from the resolver (no longer needed)

## Fix 2: getAuditLog parameter name mismatch (MEDIUM)

**File:** `src/subgraphs/audit/resolvers/audit.resolver.ts` (line 9)

**Problem:** Schema defines `getAuditLog(userId: ID!)` but the resolver destructures `{ id }`. The argument arrives as `userId` but the resolver reads `id`, so it's always `undefined`.

**Current:** `getAuditLog: async (_: any, { id }: { id: string })`

**Fix:** Change to `{ userId }: { userId: string }` and pass `userId` to the service.

---

## Fix 3: Delete unused duplicate model (MEDIUM)

**File:** `src/subgraphs/audit/models/auditLog.model.ts`

**Problem:** Defines a second `mongoose.model('AuditLog', ...)` on the same collection name. If both models are loaded, Mongoose throws `OverwriteModelError`. The file is not imported anywhere in the subgraph.

**Fix:** Delete the file. Confirmed via grep that no subgraph code imports it.

---

## Fix 4: Add metadata to AuditLog response type (MEDIUM)

**File:** `src/subgraphs/audit/schema/schema.graphql`

**Problem:** The mutation accepts `metadata: String` and the resolver stores it, but the `AuditLog` type has no `metadata` field — so stored metadata is invisible to clients.

**Fix:** Add `metadata: String` to the `AuditLog` type:
```graphql
type AuditLog @key(fields: "id") {
  id: ID!
  action: String!
  userId: ID!
  resourceId: ID!
  metadata: String
  timestamp: String!
}
```

## Fix 5: resourceId validation inconsistency (LOW)

**File:** `src/subgraphs/audit/resolvers/audit.resolver.ts` (line 24-26)

**Problem:** Schema defines `resourceId: ID` (nullable/optional), but the resolver throws if it's missing. A client sending `null` gets a runtime error rather than a schema-level error.

**Fix:** Make the validation consistent — only validate when resourceId is provided:
```ts
if (args.resourceId && !mongoose.Types.ObjectId.isValid(args.resourceId)) {
  throw new UserInputError('Invalid resourceId for audit log');
}
```
