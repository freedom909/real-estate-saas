# Tenant Switching Implementation Plan

## Overview
Implement multi-tenant switching: a user belongs to multiple tenants (via Memberships), picks an active tenant, and all subsequent requests are scoped to that tenant. The active tenant is persisted in the Session document (not in the JWT).

## Architecture Flow

```
Login → auto-select first tenant (if any) → store in Session
  ↓
Gateway reads session.activeTenantId → forwards x-tenant-id header
  ↓
Subgraphs read tenant context from x-tenant-id
  ↓
switchTenant mutation → validates membership → updates Session.activeTenantId
  ↓
Frontend stores activeTenant in zustand → refetches data
```

## Changes

### 1. Add `activeTenantId` to Session model
**File:** `src/subgraphs/auth/infrastructure/models/session.model.ts`
- Add `activeTenantId: { type: String, default: null }` to the schema

### 2. Create `switchTenant` use case
**File (new):** `src/core/tenant/application/usecases/switch-tenant.usecase.ts`
- Input: `userId`, `tenantId`
- Logic:
  1. Fetch memberships for user
  2. Verify user has a membership for the requested tenant
  3. Fetch tenant, verify it exists and is ACTIVE
  4. Return `{ tenant, membership }` (caller handles session update)

### 3. Add `switchTenant` mutation to tenant subgraph
**File:** `src/subgraphs/tenant/tenant.schema.graphql`
- Add to Mutation type:
  ```graphql
  switchTenant(tenantId: ID!): SwitchTenantPayload!
  ```
- Add payload type:
  ```graphql
  type SwitchTenantPayload {
    tenant: Tenant!
    activeTenantId: ID!
  }
  ```

**File:** `src/subgraphs/tenant/resolvers/tenant.resolver.ts`
- Add `switchTenant` resolver that:
  1. Gets userId from context (already passed via `req.user`)
  2. Calls switchTenant use case
  3. Updates the session in DB via a session repo method
  4. Returns the tenant

### 4. Add `activeTenantId` update to SessionService
**File:** `src/subgraphs/auth/infrastructure/services/session.service.ts`
- Add method: `updateActiveTenant(sessionId: string, tenantId: string): Promise<void>`
- Updates the session document in MongoDB

### 5. Add `findByUserAndTenant` to MembershipRepository
**File:** `src/core/tenant/infrastructure/repos/membership.repo.ts`
- Add: `findByUserAndTenant(userId: string, tenantId: string): Promise<MembershipDocument | null>`

### 6. Implement `tenantsByUser` resolver in user subgraph
**File:** `src/subgraphs/user/resolvers/user.resolver.ts`
- Wire up the existing schema declaration `tenantsByUser(userId: ID!): [Tenant!]!`
- Use `TenantService.getTenantsForUser()` (already exists)

### 7. Gateway: lookup session and forward `x-tenant-id`
**File:** `src/gateway/index.ts`
- In `willSendRequest`, after decoding user:
  1. Look up the session by `sessionId` from the JWT
  2. Read `session.activeTenantId`
  3. Forward as `x-tenant-id` header to subgraphs

**File:** `src/gateway/plugins/auth.plugin.ts`
- After decoding user, store `sessionId` on context (already done)
- Expose session lookup (or do it in willSendRequest)

### 8. Subgraph middleware: extract `x-tenant-id`
**File:** `src/infrastructure/auth/getUserFromContext.ts`
- Add `tenantId` to the returned context object, reading from `req.headers["x-tenant-id"]`

### 9. Frontend: tenant state store
**File (new):** `frontend/app/store/tenant.store.ts`
- Zustand store with:
  - `activeTenantId: string | null`
  - `availableTenants: Tenant[]`
  - `setActiveTenant(tenantId: string)`
  - `setAvailableTenants(tenants: Tenant[])`
- Persisted to localStorage

**File:** `frontend/app/store/auth.store.ts`
- Extend `AuthPayload.user` with optional `tenantId`

### 10. Frontend: auto-select on login
**File:** `frontend/app/auth/auth.provider.tsx`
- After login, query `tenantsByUser`
  - If 1 tenant → auto-select, call `switchTenant`
  - If 0 tenants → no tenant (customer with no org)
  - If >1 tenants → show picker (store available, don't select yet)

### 11. Frontend: tenant switcher component
**File (new):** `frontend/app/components/TenantSwitcher.tsx`
- Dropdown showing available tenants
- Calls `switchTenant` mutation on selection
- Updates tenant store
- Refetches current page data

### 12. Frontend: inject `x-tenant-id` in Apollo client
**File:** `frontend/app/lib/apollo-client.ts` (or wherever Apollo is configured)
- Add link that reads `activeTenantId` from tenant store
- Sets `x-tenant-id` header on every request
