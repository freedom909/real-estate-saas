# AI Platform Context — Wisdom Refactor

## Architecture
- **Entry**: `src/wisdom/index.ts` — Express + Apollo Server (subgraph) on port 4200
- **Dev script**: `npm run dev:ai` or `npm run dev:wisdom` (both point to `src/wisdom/index.ts`)
- **DI**: tsyringe container, registered via `src/wisdom/container/registrations/wisdom.register.ts`
- **Auth**: `getUserFromToken(req)` extracts JWT from `Authorization: Bearer <token>` header
- **Flow**: Resolver → ChatUseCase → WisdomOrchestrator → SemanticExtractor → ReferenceResolver → AgentRouter → Agent → **KnowledgeExtractor** → KnowledgeStore → WisdomResponse

## src/wisdom/ — The Intelligence Engine

### Pipeline (Knowledge Memory)
```
User Request
  → SemanticExtractor (Rule → MessageRule → LLM fallback)
  → ReferenceResolver (ordinal/pronoun resolution, auto-search)
  → AgentRouter → Agent.execute()
  → KnowledgeExtractor.extract(semantic, response, context, knowledge)
      ├── PreferenceExtractor  → PreferenceLearnedDelta[]
      ├── FactExtractor        → FactLearnedDelta[]
      ├── GoalExtractor        → GoalUpdatedDelta[]     (state machine, pure)
      └── BehaviorExtractor    → BehaviorLearnedDelta[]
  → KnowledgeStore.persist(userId, deltas)              (only Store mutates memory)
  → SummaryScheduler.onTurnComplete()                   (async, fire-and-forget)
      ├── ConversationBuffer.add(turn)
      └── if threshold exceeded:
          → SummaryAgent.summarize(turns)               (LLM compression)
          → KnowledgeStore.persist(SummaryDelta)
  → BookingStateUpdater (legacy compat)
  → WisdomResponse
```

### Memory Architecture (Knowledge Memory)

#### Core Concept: "Knowledge Deltas"
The memory system extracts **4 types of knowledge** from each conversation turn:
- `PREFERENCE_LEARNED` — what the user prefers (location, price, type, guest count) — from **user intent**
- `FACT_LEARNED` — concrete facts (search results, booking confirmations, dates) — from **system responses**
- `GOAL_UPDATED` — goal state machine transitions (NOT accumulated, one per goalType)
- `BEHAVIOR_LEARNED` — user behavior patterns (search result type consistency, etc.)

Plus async (not per-turn):
- `SUMMARY_CREATED` — compressed conversation chunks (handled by SummaryPipeline)

#### MemoryContext — Unified Key System
All memory subsystems receive a single `MemoryContext` instead of deriving keys ad-hoc:
```typescript
interface MemoryContext {
  userId: string;    // "anonymous" if not logged in
  sessionId: string; // "default" if not provided
}
```
Built once in `WisdomOrchestrator.handle()` via `buildMemoryContext(request.context.identity)`.
Passed to: `KnowledgeStore.persist(ctx, deltas)`, `KnowledgeStore.load(ctx)`, `SummaryScheduler.onTurnComplete(ctx, turn)`.

#### Goal State Machine
Goals are NOT created per-turn. One active goal per `goalType`, transitioning:
```
pending → searching → candidate_selected → booking → completed
                                                        ↘ abandoned
```
- Extractor returns `GOAL_UPDATED` delta (pure function, no mutation)
- Store applies the transition with `canTransition()` guard (never goes backward)

#### Fact vs Preference Separation
- **Preference**: user intent → "I want ibaraki", "cheap", "pet friendly"
- **Fact**: system response → search results, booking confirmations, entity details
- Rule: if it came from a search/booking response, it's a fact. If from user intent, it's a preference.

#### Storage Layers
- **Session Store** (`IMemoryStore`) — in-memory Map, per-session working memory (legacy)
- **Long-Term Store** (`ILongTermStore`) — in-memory Map (dev), Redis/DB (prod) — holds `IUserKnowledge`
- **KnowledgeStore** — applies `KnowledgeDelta[]` → `ILongTermStore` (only mutator)

### Plugin Architecture (Extractor Side)
```
KnowledgeExtractor (66 lines, thin orchestrator)
  └── @injectAll(KNOWLEDGE_PLUGIN_TOKEN) plugins: KnowledgePlugin[]
      ├── PreferencePlugin  — user intent → PreferenceLearnedDelta[]
      ├── FactPlugin        — system response → FactLearnedDelta[]
      ├── GoalPlugin        — state machine → GoalUpdatedDelta[]
      └── BehaviorPlugin    — interaction patterns → BehaviorLearnedDelta[]
```
- **Interface**: `KnowledgePlugin.extract(semantic, response, context, knowledge) → KnowledgeDelta[]`
- **Adding new memory type**: create plugin class → register under `KNOWLEDGE_PLUGIN_TOKEN` → done. No changes to KnowledgeExtractor.

### Handler Architecture (Store Side)
```
KnowledgeStore (90 lines, dispatcher)
  └── @injectAll(deltaHandlers) handlers: Map<DeltaKind, DeltaHandler>
      ├── PreferenceHandler  — dedup by key+value
      ├── FactHandler        — dedup by key
      ├── GoalHandler        — state machine with canTransition()
      ├── BehaviorHandler    — stored as fact with "behavior:" prefix
      └── SummaryHandler     — bounded list (max 20)
```
- **Interface**: `DeltaHandler.apply(knowledge, data)` — mutates knowledge in-place
- **Adding new memory type**: create handler → register under `WISDOM_TOKENS.memory.deltaHandlers` → done. Store picks it up automatically.
- **DI**: `@injectAll(WISDOM_TOKENS.memory.deltaHandlers)` collects all handlers, builds `Map<kind, handler>` in constructor.

### Design Rules
1. **Extractors are pure functions** — given inputs, produce deltas. No side effects, no state mutation.
2. **Only KnowledgeStore modifies memory** — Extractor never touches `knowledge.goals[]`, `knowledge.facts[]`, etc.
3. **No summary per turn** — SummaryPipeline (async) handles compression periodically.
4. **Goal state machine** — one active goal per type, never accumulates duplicates.
5. **Fact vs Preference** — user intent = Preference, system response = Fact.
6. **Plugin architecture** — KnowledgeExtractor is a thin orchestrator. New memory types = new plugins.
7. **Handler architecture** — KnowledgeStore is a thin dispatcher. New memory types = new handlers.
8. **Summary is async** — ConversationBuffer → SummaryScheduler → SummaryAgent → KnowledgeStore. Never blocks the per-turn response.
